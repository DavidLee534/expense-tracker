# 발표 대본 — 연간 지출 관리 (expense-tracker)

`expense-tracker-portfolio.pptx` 17개 슬라이드 순서에 맞춘 발표 대본입니다.
전문 용어를 최대한 줄이고, 처음 듣는 분도 이해할 수 있게 쉬운 말로 풀어 썼습니다.
괄호 안은 발표 시 참고할 진행 메모입니다.

---

## 1. 타이틀

안녕하세요. 오늘 소개해드릴 건 **"연간 지출 관리"**라는 가계부 웹사이트입니다.
쉽게 말하면, 내가 쓴 돈을 기록하고 한눈에 볼 수 있게 해주는 서비스예요. 사용자는 두 종류가 있는데, 하나는 지출을 직접 기록하는 "고객"이고, 다른 하나는 회원들과 서비스 전체를 관리하는 "관리자"입니다.
처음에는 화면만 있는 간단한 형태로 만들었다가, 나중에 진짜 데이터베이스(정보를 저장하는 창고라고 생각하시면 됩니다)를 연결해서 실제로 쓸 수 있는 서비스에 가깝게 만들었어요. 그 과정을 순서대로 설명드리겠습니다.

## 2. 기획 의도

왜 이걸 만들었냐면요, 지출을 기록해도 여기저기 흩어져 있으면 "이번 달에 뭐에 얼마 썼는지" 파악하기가 어렵잖아요. 그 불편함을 해결해보고 싶었습니다.
그래서 세 가지를 목표로 잡았어요.
첫째, 아이디어 구상부터 화면 설계, 실제 코드 작성, 그리고 배포까지 — 처음부터 끝까지 전 과정을 직접 다 해보는 것.
둘째, "고객용 화면"과 "관리자용 화면"을 다르게 만들고, 각자 볼 수 있는 정보도 다르게 제한해보는 것.
셋째, 단순히 지출만 기록하는 게 아니라 사람들끼리 소통할 수 있는 게시판도 추가해서, 사람들이 계속 방문하고 싶은 서비스로 만들어보는 것이었습니다.

## 3. 프로젝트 개요

이 서비스는 지출을 쉽게 기록하고, 카테고리별·기간별로 통계를 보여줘서 "내가 돈을 어디에 많이 쓰는지" 바로 알 수 있게 해줍니다.
앞서 말씀드린 문제(흩어진 데이터)를 풀기 위해, 등록은 최대한 간단하게 만들고 통계는 도넛 모양이나 막대 그래프 같은 그림으로 보여줘서 숫자만 보는 것보다 훨씬 이해하기 쉽게 했어요.
그리고 고객은 자기 지출만 기록하고 확인하고, 관리자는 회원 목록과 서비스 전체 통계를 관리하도록 역할을 나눴습니다.

## 4. 핵심 기능 — 고객

고객이 할 수 있는 걸 하나씩 말씀드릴게요.
먼저 회원가입하고 로그인할 수 있고요, 지출은 등록하고, 보고, 고치고, 지울 수 있습니다. 날짜나 카테고리로 걸러서 볼 수 있고, 메모로 검색도 됩니다.
지출이 많이 쌓이면 목록이 끝없이 길어지니까, 한 페이지에 10개씩만 보이게 나눠서 보여주는 기능도 넣었어요. 필터를 바꾸면 자동으로 첫 페이지로 돌아갑니다.
그리고 이번 달·이번 해 통계를 도넛 그래프나 막대 그래프로 바꿔가면서 볼 수 있고, 카테고리는 식비·교통·주거처럼 8가지로 미리 정해놨어요.
마지막으로 이번에 새로 만든 게시판이 있는데, 여기서 절약 팁을 자유롭게 쓰고 댓글도 달 수 있어요. 다만 내가 쓴 글이나 댓글만 지울 수 있게 만들었습니다.

## 5. 핵심 기능 — 관리자

관리자는 회원가입 없이, 정해진 계정으로만 들어올 수 있습니다.
들어오면 전체 회원 수, 전체 지출 합계, 이번 달 지출, 그리고 사람들이 제일 많이 쓰는 카테고리 5개를 한눈에 볼 수 있는 화면이 나와요. 회원 관리 화면에서는 이름이나 이메일로 검색하고, 문제 있는 계정은 정지하거나 다시 풀어줄 수 있습니다.
이번에 추가한 기능으로, 게시판에 올라온 글도 검색해서 지울 수 있게 했어요.
근데 관리자는 지출을 직접 등록하지는 않아요. 어디까지나 "확인하고 관리하는 역할"만 하도록 딱 나눠뒀습니다.

## 6. 스크린샷 (실제 화면)

말로 설명하는 것보다 실제 화면을 보여드리는 게 빠르겠죠. (화면 넘기며) 고객이 보는 대시보드, 페이지가 나눠진 지출 목록, 그래프로 보는 통계, 그리고 관리자 화면입니다. 전부 실제로 앱을 켜서 찍은 화면이에요.

## 7. 기술 스택

이 서비스를 어떤 도구로 만들었는지 간단히 말씀드릴게요. 크게 네 가지로 나눌 수 있어요.
화면을 만드는 부분은 특별한 도구 없이 HTML, CSS, 자바스크립트라는 가장 기본적인 웹 기술만 썼습니다. 색상이나 여백 같은 디자인 값은 한 곳에서 관리하도록 정리했고요.
데이터를 저장하는 부분은 Supabase라는 서비스를 썼는데, 쉽게 말하면 "데이터베이스를 대신 관리해주는 서비스"예요. 로그인 기능도 여기서 제공하는 걸 그대로 가져다 썼고, 각 사용자가 자기 데이터만 볼 수 있게 막는 규칙도 여기서 설정했습니다.
협업은 깃허브를 이용해서, 기능을 나눠서 작업하고 검토한 다음 합치는 방식으로 진행했고요.
마지막으로, 실제로 화면이 잘 동작하는지 자동으로 확인하는 도구, 이 발표자료 자체를 자동으로 만들어주는 도구, 그리고 데이터 구조를 그림으로 그려주는 도구도 함께 사용했습니다.

## 8. 설계 원칙

코드를 짤 때 지킨 원칙이 네 가지 있어요.
첫째, 한 화면에 필요한 파일(구조, 디자인, 동작)을 전부 같은 폴더에 모아뒀습니다. "이 기능 고치려면 어디를 봐야 하지?"라는 고민을 없애기 위해서예요.
둘째, 고객이 쓰는 화면과 관리자가 쓰는 화면을 폴더 자체를 완전히 나눴습니다. 다만 데이터를 다루는 공통 코드는 같이 씁니다.
셋째, 주소창 경로만 봐도 무슨 화면인지 알 수 있게 이름을 지었어요.
넷째, 카테고리처럼 잘 안 바뀌는 데이터는 아예 손대지 못하게 고정해서 데이터가 뒤섞이지 않게 했습니다.

## 9. 데이터 모델 (ERD)

이건 데이터가 어떻게 연결돼 있는지 보여주는 그림이에요. 회원 정보가 중심에 있고, 거기서 지출 기록, 그리고 이번에 추가한 게시글·댓글로 선이 이어집니다. 카테고리는 지출 기록에서 골라 쓰는 고정된 목록이고요.
이 그림은 실제 문서(README)에 있는 그림과 완전히 똑같은 걸 그대로 가져온 거라, 문서랑 발표자료가 서로 다른 내용을 말하지 않도록 맞춰뒀습니다.

## 10. 기술 스택 진화 (localStorage → Supabase)

처음에는 서버도 없이, 그냥 내 브라우저 안에만 데이터를 저장하는 방식이었어요. 심지어 비밀번호도 그대로 저장했었죠, 어디까지나 연습용이었으니까요. 문제는 다른 기기나 다른 브라우저로 들어가면 아무 데이터도 안 보인다는 거였어요.
그래서 Supabase로 옮기면서, 로그인 기능은 Supabase가 제공하는 걸 그대로 써서 비밀번호를 직접 다룰 필요를 없앴고, 회원·카테고리·지출 정보를 저장할 표(테이블)를 만들었고, 각자 자기 데이터만 볼 수 있게 막는 규칙도 걸었습니다.

## 11. 보안 & 데이터 무결성

보안 쪽에서 신경 쓴 부분을 쉽게 설명드릴게요.
먼저, 고객은 자기 데이터만 보이고 관리자만 전체를 볼 수 있게, 데이터베이스 자체에 규칙을 걸어뒀습니다. 화면 코드가 아니라 데이터베이스 단에서 막아두는 거라 더 안전해요.
근데 "이 사람이 관리자인지 확인하는 규칙"을 만들다가 예상치 못한 문제가 하나 생겼는데, 이건 잠시 후 트러블슈팅에서 말씀드릴게요.
그리고 Supabase가 제공하는 보안 점검 도구로 한 번 더 확인해서, 굳이 외부에 열려있을 필요가 없는 기능들을 닫아뒀고, 비밀번호는 직접 다루지 않고 전문 서비스(Supabase Auth)에 맡겨서 실수할 여지를 아예 없앴습니다.

## 12. 협업 워크플로우

작업은 기능 단위로 나눠서 진행했어요. 공통으로 쓰는 부분 먼저, 그다음 로그인 기능, 그다음 핵심 기능 순서로 작업하고, 다 만들면 검토를 거쳐서 원본에 합쳤습니다.
합치고 나서 더는 안 쓰는 작업 브랜치는 바로바로 지워서 저장소를 깔끔하게 유지했어요.

## 13. 시연

이제 직접 보여드릴게요. 순서는 이렇습니다.
먼저 회원가입하고 바로 로그인되는 걸 보여드리고요, 지출을 몇 개 등록한 다음 목록에서 필터랑 페이지 나누기가 어떻게 동작하는지 보여드릴게요.
그다음 통계 화면에서 그래프 모양을 바꿔보고, 게시판에 절약 팁을 하나 써보고 댓글도 달아보겠습니다.
마지막으로 관리자 계정으로 들어가서 전체 통계를 보고, 회원을 정지·해제해보고, 게시판 글도 하나 지워보면서 관리 기능까지 보여드리겠습니다.

*(여기서 실제 화면으로 전환해 라이브 데모 진행)*

## 14. 문제 해결 (트러블슈팅)

작업하면서 실제로 겪었던 문제들을 공유해볼게요.
첫째, "이 사람이 관리자인지 확인하는 규칙"을 만드는데, 그 규칙 안에서 회원 정보를 다시 확인하려다 보니 규칙이 자기 자신을 계속 부르는 문제가 생겼어요. 마치 거울을 두 개 마주 보게 놓은 것처럼 끝없이 반복되는 상황이었죠. 이건 별도의 "특별 권한을 가진 함수"를 하나 만들어서 그 안에서만 확인하도록 바꿔서 해결했습니다.
둘째, 보안 점검 도구를 돌려봤더니, 방금 그 특별 함수가 일반 사용자한테도 그냥 열려있다는 경고가 나왔어요. 필요 없는 권한을 회수해서 막았습니다.
셋째, 발표자료용 스크린샷을 찍는데 화면에 "이번 달 지출 0원"이라고 나오는 이상한 상황이 있었어요. 알고 보니 테스트용 가짜 데이터를 만드는 코드에서 날짜 계산이 한 달 어긋나 있었던 거예요. 날짜 계산을 다시 고치고 스크린샷도 새로 찍었습니다.
넷째, 테스트하려고 로컬로 서버를 띄워놓고 다른 작업을 하면 서버가 갑자기 꺼지는 문제가 있었는데, 서버를 띄운 터미널 창이 닫히면서 서버도 같이 꺼진 거였어요. 서버가 터미널과 상관없이 계속 켜져 있도록 설정을 바꿔서 해결했습니다.

## 15. 개선점

앞으로 더 좋게 만들고 싶은 부분은 이렇습니다.
지금 관리자 로그인은 미리 정해둔 계정 하나로만 되는데, 이것도 고객처럼 정식 로그인 시스템으로 바꾸는 걸 고민하고 있어요.
그리고 지금 고객 화면은 여전히 브라우저에만 데이터를 저장하는 옛날 방식으로 동작하는데, 이걸 실제 데이터베이스와 연결하는 게 다음 목표입니다.
게시판에는 이상한 글을 신고하거나 차단하는 기능을 추가해서 더 안전하게 관리하고 싶고, 관리자 화면에는 실시간으로 바뀌는 통계나 알림 기능도 넣어보고 싶습니다.

## 16. 회고 — 배운 점

이번에 가장 크게 느낀 건, 연습용으로 만든 걸 실제로 쓸 수 있게 바꾸려면 로그인이나 보안을 처음부터 다시 고민해야 한다는 점이었어요.
그리고 데이터베이스 규칙을 만들다가 생긴 반복 문제를 해결하는 방법도 배웠고, 보안 점검 도구로 눈에 잘 안 보이는 부분까지 확인하는 습관도 생겼습니다.
