# 발표 대본 — 연간 지출 관리 (expense-tracker)

`expense-tracker-portfolio.pptx` 17개 슬라이드 순서에 맞춘 발표 대본입니다.
괄호 안은 발표 시 참고할 진행 메모입니다.

---

## 1. 타이틀

안녕하세요. 오늘 소개해드릴 프로젝트는 **"연간 지출 관리"**, 개인의 지출을 기록하고 관리자가 회원과 서비스 전체 통계를 관리할 수 있는 지출 관리 웹 서비스입니다.
Vanilla JS와 HTML/CSS로 시작해서, 실제 백엔드인 Supabase(PostgreSQL)까지 연동한 프로젝트예요. 지금부터 기획 의도부터 트러블슈팅, 앞으로의 개선점까지 순서대로 말씀드리겠습니다.

## 2. 기획 의도

이 프로젝트를 시작한 배경은 간단합니다. 지출을 기록해도 데이터가 흩어져 있으면 "이번 달에 뭐에 얼마를 썼는지" 한눈에 파악하기가 어렵다는 점이었어요.
그래서 세 가지 목표를 세웠습니다.
첫째, 기획부터 설계, 구현, 배포까지 — 정적 프로토타입에서 실제 백엔드 연동까지 전체 사이클을 직접 경험해보는 것.
둘째, 고객과 관리자라는 서로 다른 역할에 맞춰 화면과 데이터 접근 권한을 다르게 설계하는 연습.
셋째, 지출 관리라는 단일 기능에 게시판을 더해서 사용자 리텐션까지 고려한 기능 확장을 해보는 것이었습니다.

## 3. 프로젝트 개요

이 서비스는 개인이 지출을 손쉽게 기록하고, 카테고리별·기간별 통계로 소비 패턴을 파악할 수 있게 도와줍니다.
문제 정의는 앞서 말씀드린 대로 "흩어진 데이터" 문제고요, 핵심 가치는 간편한 등록 흐름과 도넛·막대 차트 같은 시각적 통계로 소비 패턴을 즉시 인지할 수 있다는 점입니다.
그리고 고객과 관리자, 두 가지 역할로 나눠서 고객은 본인 지출을 기록·조회하고, 관리자는 회원과 서비스 전체 통계를 관리하도록 설계했습니다.

## 4. 핵심 기능 — 고객

고객이 사용할 수 있는 기능을 말씀드리면, 먼저 회원가입과 로그인이 있고요, 지출은 등록·조회·수정·삭제가 모두 가능합니다. 기간이나 카테고리로 필터링하고 메모로 검색도 할 수 있어요.
지출 내역이 많아지는 걸 대비해서 페이지네이션도 넣었습니다. 페이지당 10건씩 보여주고, 필터를 바꾸면 자동으로 1페이지로 돌아가요.
그리고 월별·연별 통계를 도넛 차트와 막대 차트로 토글해서 볼 수 있고, 카테고리는 식비·교통·주거 등 8종으로 고정해서 모든 사용자가 동일하게 사용합니다.
마지막으로 이번에 추가한 게시판 기능도 있는데, 절약 팁을 자유롭게 작성하고 댓글로 소통할 수 있고, 본인이 쓴 글이나 댓글만 삭제할 수 있게 권한을 제한했습니다.

## 5. 핵심 기능 — 관리자

관리자는 회원가입 없이 고정된 계정으로만 로그인합니다.
로그인하면 전체 회원 수, 총 지출액, 이번 달 지출액, 인기 카테고리 TOP 5를 한눈에 볼 수 있는 대시보드가 있고요, 회원 관리 화면에서는 이름이나 이메일로 검색하고 계정을 정지하거나 해제할 수 있습니다.
이번에 추가한 게시판 관리 기능으로 부적절한 글도 검색해서 삭제할 수 있게 했습니다.
다만 관리자는 지출 데이터를 직접 등록하지는 않아요. 어디까지나 조회와 관리만 담당하도록 역할을 분리했습니다.

## 6. 스크린샷 (실제 화면)

말로만 설명하는 것보다 실제 화면을 보여드리는 게 빠르겠죠. (화면 넘기며) 고객 대시보드, 페이지네이션이 적용된 지출 내역, 도넛·막대 차트로 보는 통계, 그리고 관리자 대시보드입니다. 실제로 앱을 구동해서 캡처한 화면이에요.

## 7. 기술 스택

기술 스택은 크게 네 영역으로 나눠서 말씀드릴게요.
프론트엔드는 프레임워크 없이 HTML5, CSS3, 순수 자바스크립트로 만들었습니다. CSS는 변수 기반으로 디자인 시스템을 만들어서 일관성을 유지했고요.
백엔드와 DB는 Supabase 위에서 PostgreSQL 17을 쓰고, 인증은 Supabase Auth, 데이터 접근 제어는 Row Level Security로 처리했습니다.
협업과 배포는 Git과 GitHub, GitHub CLI로 PR 기반 브랜치 전략을 따랐고요.
검증과 문서화 쪽은 Playwright로 실제 브라우저 시나리오를 검증하고, python-pptx로 이 발표자료 자체를 자동 생성했고, ERD는 Mermaid로 그렸습니다.

## 8. 설계 원칙

설계 원칙은 네 가지입니다.
첫째, 코로케이션 — 각 페이지의 HTML, CSS, JS를 같은 폴더에 평탄하게 배치해서 "이 기능을 고치려면 어느 폴더를 보면 되는가"에 바로 답할 수 있게 했습니다.
둘째, 역할별 분리 — 고객은 루트 경로, 관리자는 /admin 하위로 완전히 나누되 데이터 계층은 공유하도록 했고요.
셋째, 클린 URL — list, create, edit처럼 목적이 드러나는 경로를 썼습니다.
넷째, 카테고리 같은 고정 공통 데이터는 CRUD 대상으로 두지 않고 참조 데이터로 취급해서 데이터 일관성을 지켰습니다.

## 9. 데이터 모델 (ERD)

데이터 모델은 이렇게 구성했습니다. profiles가 중심이 되고, 여기서 expenses, 그리고 이번에 추가한 posts와 comments로 연결됩니다. categories는 expenses에 참조되는 고정 테이블이고요.
이 다이어그램은 실제로 README.md에 있는 mermaid 코드를 그대로 렌더링한 거라, 문서와 발표자료의 ERD가 항상 같은 내용을 보여주도록 맞춰뒀습니다.

## 10. 기술 스택 진화 (localStorage → Supabase)

원래는 백엔드 서버 없이 브라우저의 localStorage에 데이터를 저장하는 정적 사이트였습니다. 심지어 비밀번호도 평문으로 저장했었어요, 데모 목적이었으니까요. 문제는 브라우저나 기기를 바꾸면 데이터가 공유되지 않는다는 점이었죠.
그래서 Supabase로 옮기면서 인증은 Supabase Auth에 위임해서 평문 비밀번호 문제를 없앴고, profiles·categories·expenses 테이블을 설계했고, Row Level Security로 행 단위 접근 제어를 걸었습니다.

## 11. 보안 & 데이터 무결성

보안 쪽에서 신경 쓴 부분을 말씀드리면, 먼저 Row Level Security로 고객은 본인 데이터만, 관리자는 role 기반으로 전체 데이터에 접근하도록 했습니다.
그런데 관리자 여부를 판별하는 정책을 짜다 보니 재귀 문제가 생겼는데, 이건 잠시 후 트러블슈팅에서 자세히 말씀드릴게요.
그리고 Supabase의 Security Advisor로 점검해서, 내부 함수가 불필요하게 외부에 노출되는 권한을 회수했고, 비밀번호 저장·검증은 직접 구현하지 않고 Supabase Auth에 위임해서 자체 구현 리스크를 없앴습니다.

## 12. 협업 워크플로우

협업은 기능 단위로 브랜치를 나눠서 진행했습니다. 공통 자원(feat/shared) → 인증(feat/auth) → 핵심 기능(feat/core) 순으로 작업하고, Pull Request로 main에 병합했어요.
병합이 끝난 브랜치나 더 이상 쓰지 않는 브랜치는 바로바로 삭제해서 원격 저장소를 깔끔하게 유지했습니다.

## 13. 시연

이제 직접 시연해보겠습니다. 순서는 이렇습니다.
먼저 회원가입하고 바로 로그인이 되는 걸 보여드리고요, 지출을 몇 건 등록한 다음 목록에서 필터랑 페이지네이션이 어떻게 동작하는지 보여드릴게요.
그 다음 통계 페이지에서 도넛 차트와 막대 차트를 토글해보고, 게시판에 절약 팁을 하나 작성하고 댓글도 달아보겠습니다.
마지막으로 관리자 계정으로 로그인해서 전체 통계를 보고, 회원을 정지·해제해보고, 게시판 글도 하나 삭제해보면서 모더레이션 기능까지 보여드리겠습니다.

*(여기서 실제 화면으로 전환해 라이브 데모 진행)*

## 14. 문제 해결 (트러블슈팅)

작업하면서 실제로 겪었던 문제 몇 가지를 공유하겠습니다.
첫째, RLS 무한 재귀입니다. profiles 테이블의 정책 안에서 "이 사람이 관리자인가"를 확인하려고 profiles.role을 다시 조회하게 만들었더니, 정책이 정책을 부르면서 무한 재귀가 발생했어요. 이건 SECURITY DEFINER로 만든 is_admin() 함수를 따로 두어 우회하는 방식으로 해결했습니다.
둘째, Supabase의 Security Advisor를 돌려봤더니 방금 말씀드린 is_admin 같은 내부 함수가 일반 사용자도 REST API로 직접 호출할 수 있게 노출돼 있다는 경고가 나왔습니다. 불필요한 EXECUTE 권한을 회수해서 해결했고요.
셋째, 포트폴리오용 스크린샷을 찍는데 대시보드에 "이번 달 지출 0원"으로 나오는 버그가 있었습니다. 알고 보니 더미 데이터를 시딩하는 스크립트의 월 계산 로직이 한 달 어긋나 있었던 거예요. 로직을 오늘 날짜 기준으로 고치고 다시 캡처해서 해결했습니다.
넷째, 로컬 서버를 띄우고 다음 명령을 실행하면 서버 연결이 끊기는 문제가 있었는데, 이건 서브셸에서 실행한 프로세스가 셸이 끝나면서 같이 죽었던 거라 nohup과 disown으로 완전히 분리해서 해결했습니다.

## 15. 개선점

앞으로 개선하고 싶은 부분은 이렇습니다.
지금은 관리자 로그인이 하드코딩된 고정 계정인데, 이것도 Supabase Auth 기반으로 전환하는 걸 검토하고 있고요.
프론트엔드는 아직 localStorage를 쓰고 있는데, 이걸 실제로 Supabase 클라이언트 호출로 교체하는 작업이 다음 단계입니다.
게시판에는 신고나 차단 기능을 추가해서 커뮤니티 모더레이션 수준을 높이고 싶고, 관리자 대시보드에는 실시간 통계나 알림 기능을 붙여보고 싶습니다.

## 16. 회고 — 배운 점

이번 프로젝트에서 가장 크게 배운 건, 정적 데모에서 실제 백엔드로 옮기는 과정에서 인증과 보안을 처음부터 다시 설계해야 한다는 점이었어요.
그리고 RLS 정책에서 재귀 문제를 SECURITY DEFINER 패턴으로 해결하는 방법을 익혔고, Security Advisor 같은 도구로 눈에 보이지 않는 권한 노출까지 점검하는 습관을 들이게 됐습니다.
